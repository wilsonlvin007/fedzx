# 技术文档 01｜整体架构与代码结构（fedzx.com）

本文面向接手开发者与 AI，目标是让你在 30 分钟内理解系统全貌与关键约束，并能独立定位问题、迭代功能与上线。

---

## 1. 系统概览

`fedzx.com` 当前是「静态内容站 + CMS（Next.js）」的组合架构：

- 前台站点（静态）：`https://fedzx.com`
  - 由 Nginx 直接从磁盘目录 `/var/www/fedzx` 提供静态文件（HTML/CSS/JS）
  - 页面通过浏览器端 JS 调用 CMS 公共 API 获取文章数据并渲染
- CMS（动态）：`https://cms.fedzx.com`
  - Next.js 16（App Router）
  - 后台管理：文章 / 页面模块 / 推荐位 / 导入与 AI 草稿（预留）
  - 数据库：SQLite（生产单机部署，便于维护与备份）

整体目标是：

- Phase 1：运营手动维护（已完成）
- Phase 2：半自动导入（RSS/API + AI 草稿 + 人审）
- Phase 3：自动运营（Agent pipeline）

---

## 2. 仓库结构（Monorepo）

仓库根目录（服务器 `/opt/fedzx`，本地同名仓库）：

- `apps/site/`
  - 纯静态站（HTML + JS）
  - 通过 `meta[name="fedzx-cms-base"]` 指定 CMS Base URL（生产为 `https://cms.fedzx.com`）
- `apps/cms/`
  - Next.js 16 CMS 后台 + API
  - Prisma ORM + SQLite
  - Admin 登录基于 session cookie
- `docs/`
  - 知识库 Markdown 文件（服务器路径同为 `/opt/fedzx/docs`）
  - 由 CMS 内部 docs API 读取并供静态站 `/docs/*` 展示
- `deploy/`
  - Nginx 与 systemd 的参考配置（服务器通常会拷贝或手工同步到 `/etc/nginx/...` 和 `/etc/systemd/...`）

## 3. 运行形态（生产环境）

### 3.1 域名与端口

- `fedzx.com / www.fedzx.com`：
  - Nginx 直接服务静态文件
  - HTTPS（Let’s Encrypt）
- `cms.fedzx.com`：
  - Nginx 反向代理到 `127.0.0.1:3100`
  - HTTPS（同一张证书或包含该子域的证书）
- CMS 服务：
  - systemd 服务名：`fedzx-cms`
  - 监听端口：`3100`

### 3.2 文件与路径

- 静态站根目录：`/var/www/fedzx`
- 仓库：`/opt/fedzx`
- CMS 生产环境变量：`/opt/fedzx/apps/cms/.env.production`
- CMS 数据库（SQLite）：`/opt/fedzx/apps/cms/prod.db`

---

## 4. 内容如何流动（从 CMS 到前台）

### 4.1 CMS 内容源

文章由 CMS 后台创建与发布（status=Published），写入 SQLite。

### 4.2 公共 API（前台消费）

静态站通过浏览器 `fetch` 请求：

- 文章列表：`GET https://cms.fedzx.com/api/public/articles`
- 文章详情：`GET https://cms.fedzx.com/api/public/articles/<slug>`

前台的关键约束：

- 前台是纯静态，不能依赖服务端渲染（SSR）
- 数据完全来自 API（页面结构固定，数据动态填充）

## 5. 知识库 Docs 系统（内部）

Docs 是“静态站展示 + CMS 读取 Markdown”组合：

- Markdown 存在服务器：`/opt/fedzx/docs/`
  - 以子目录分类：`01-运营文档/`, `02-技术文档/` 等
  - 文件名即 slug，例如 `agent-roles.md` -> slug `agent-roles`
  - Markdown 第一行 `# ...` 会作为标题（目录与正文标题都读这个）
- CMS 内部 API：
  - `GET /api/internal/docs`：返回分类目录与文档列表
  - `GET /api/internal/docs/<slug>`：返回 `{title, content, slug, category}`
- 前台页面：
  - `/docs/`：目录页，调用 `GET /api/internal/docs`
  - `/docs/<slug>/`：文档页，调用 `GET /api/internal/docs/<slug>`
  - 文档渲染为自研轻量 Markdown 渲染（避免依赖外网 CDN 模块）

注意：

- docs API 有 10 秒缓存（进程内缓存），新增/修改 Markdown 后一般 10 秒内生效。
- 如需立刻生效，可 `sudo systemctl restart fedzx-cms`。

## 6. 本地开发（建议流程）

### 6.1 CMS（Next.js）

在 `apps/cms`：

- 安装依赖：`npm ci`
- 初始化数据库（dev）：`npm run db:init`
- 数据库迁移（dev）：`npm run db:migrate`
- 启动：`npm run dev`

### 6.2 静态站（无需构建）

`apps/site` 是静态目录，你可以用任意静态服务器预览：

- 例如用 `python3 -m http.server` 或 VSCode Live Server
- 注意：本地预览时需要 CMS 也在本地运行，或把 `meta fedzx-cms-base` 指到线上 CMS

## 7. 关键设计取舍（为什么这样做）

- 静态站 + API：
  - 运维成本低，Nginx 直接服务，性能稳定
  - SEO 基础仍可保证（静态 HTML + 可访问路径 + 内链）
  - 数据更新不需要重新构建前台
- SQLite（Phase 1/2）：
  - 单机最简单，迁移/备份明确
  - 未来 Phase 4+ 可升级到 Postgres/Supabase（需要改 Prisma datasource 与迁移策略）
- Docs 内部系统：
  - 文档只要“上传 md 文件”即可自动出现
  - 不依赖外部 Markdown 渲染库，避免网络与供应链问题
